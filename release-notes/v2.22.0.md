## v2.22.0

> **Upgrade the server before the desktop and mobile clients.**
> This release adds the `e2ee_enabled` flag to the server list and the WS ready payload, and the
> client now refuses to send when it cannot tell whether a conversation is encrypted — sending
> plaintext to a server that mandates encryption is the one outcome worth blocking outright.
> Against a server older than v2.22.0 the field is simply absent, so clients will report that they
> cannot determine the encryption state and will not send. Upgrading the server first avoids this
> entirely; no data migration is involved.

- **uploads**: byte-accurate progress and a cancel that actually aborts the transfer
- **uploads**: client size limit aligned with the server's and with Cloudflare's edge cap
- **uploads**: files rejected for size or type now say so instead of disappearing
- **attachments**: chat images and videos send a thumbnail, so opening a channel no longer downloads
  every picture at full size
- **attachments**: videos carry a poster frame; the video itself is untouched until playback
- **avatars, icons, banners**: downscaled on the client instead of uploading a full-resolution PNG
- **feedback**: video attachments accepted alongside images
- **mobile**: attachments picked through the native Android picker, which is what lets a video the
  browser cannot decode still get a poster

- **e2ee**: the server now enforces its own encryption policy — a message must match the
  conversation's setting in both directions, on send and on edit
- **e2ee**: editing a message across an encryption toggle keeps the edit instead of dropping it
- **e2ee**: encrypted attachments show their preview without downloading the original
- **search**: messages stay searchable after a conversation is switched back to plaintext
- **storage**: thumbnails are charged to the uploader's quota and released with the message
